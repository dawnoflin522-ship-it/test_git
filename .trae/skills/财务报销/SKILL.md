# 财务报销审核流程

本skill用于处理员工的财务报销申请，通过与财务报销规则比对，判断申请是否符合要求。

## 触发条件

当用户有以下表述时触发：
- "我要报销"
- "帮我处理报销"
- "申请费用报销"
- "这个发票能报销吗"
- "差旅费报销"
- "餐饮费报销"
- 任何与费用报销相关的请求

## 报销信息收集流程

### 第一步：收集必要信息

需要向用户收集以下5项信息，如果用户未提供某一项，必须追问：

1. **报销金额** - 具体报销的数额（元）
2. **报销项目** - 费用类型（如：差旅费、餐饮费、住宿费、交通费、办公用品费等）
3. **所属部门和职位** - 用户所在的部门名称和职位
4. **报销理由** - 报销的具体原因和业务背景
5. **发票图片** - 用户需要上传报销发票的图片

**追问策略**：
- 每次只追问缺失的信息，不要一次性追问所有
- 使用礼貌、专业的语气
- 示例："请提供您的所属部门和职位，以便我核对报销权限。"

### 第二步：规则比对

当5项信息全部收集完成后：

1. 读取 [财务报销规则](references/财务报销规则.md)
2. 根据用户的部门和职位，查找对应的报销限额和权限
3. 比对报销金额是否在允许范围内
4. 比对报销项目是否属于该部门/职位可报销的范围
5. 判断报销理由是否合理

**比对结果**：
- 如果符合规则要求 → 进入第三步（发票OCR识别）
- 如果不符合规则要求 → 返回拒绝报销的回复，说明具体原因

### 第三步：发票OCR识别

使用 `mineru_mcp_server` MCP工具对发票图片进行OCR识别：

```
工具：mcp_mineru-mcp_parse_documents
参数：
- file_sources: 用户上传的发票图片路径
- enable_ocr: true
- language: "ch"
```

**识别内容**：
- 发票金额
- 发票项目/服务名称
- 发票日期
- 开票单位

### 第四步：信息一致性校验

将OCR识别结果与用户提供的信息进行比对：

| 比对项 | 要求 |
|--------|------|
| 金额 | OCR识别的金额应与用户提供的报销金额一致（允许±0.1元误差） |
| 项目 | OCR识别的服务名称应与报销项目类型相符 |

**校验结果**：
- 如果信息一致 → 返回同意报销的回复
- 如果信息不一致 → 返回拒绝报销的回复，说明差异点

## 回复格式

### 同意报销回复模板

```
✅ 报销申请审核通过

【申请信息】
- 报销金额：XXX元
- 报销项目：XXX
- 所属部门：XXX
- 职位：XXX
- 报销理由：XXX

【审核结果】
✓ 符合部门报销权限
✓ 金额在限额范围内
✓ 发票信息核验通过

您的报销申请已通过初步审核，请按公司流程提交纸质材料至财务部门。
```

### 拒绝报销回复模板

```
❌ 报销申请审核未通过

【申请信息】
- 报销金额：XXX元
- 报销项目：XXX
- 所属部门：XXX
- 职位：XXX

【拒绝原因】
[具体说明不符合的规则条款或信息不一致的地方]

如有疑问，请咨询财务部门。
```

## 异常处理

### 一般异常处理

如果在处理过程中遇到**除MCP服务异常以外**的任何异常：
- 告知用户："财务报销服务异常，请稍后再试。"
- 不要暴露具体的技术错误信息

### MCP服务异常处理

如果使用 `mineru_mcp_server` 进行OCR识别时发生异常：

1. **第一次重试**：等待2秒后重新调用
2. **第二次重试**：等待3秒后再次重新调用
3. **重试均失败**：告知用户"发票识别服务暂时不可用，请稍后再试或联系财务人员人工处理。"

## 注意事项

1. **信息完整性**：必须确保5项信息全部收集完成才能进入规则比对
2. **规则准确性**：严格按照reference文件中的规则执行，不得主观判断
3. **隐私保护**：处理过程中涉及的员工信息不得记录或泄露
4. **专业态度**：保持专业、礼貌的沟通方式
5. **异常处理**：遇到异常时给用户清晰的反馈，不要暴露技术细节
